# 5.3 Merging and Splitting Flows – Network Analysis

## 5.3.1 From Serial Lines to Networks
In previous sections, systems were **pure serial lines**.
In this chapter we generalize the analysis to **networks**, where flows may:
- split into parallel branches,
- merge downstream,
- interact in non-linear ways.

The key principle remains:
> analyze each workstation separately, but **track flow fractions and variability propagation**.

---

## 5.3.2 Fundamental Rule for Networks
For any workstation:
- arrivals are characterized by:
  - arrival rate λ
  - arrival variability C_A²
- departures generate:
  - new arrival parameters for downstream stations

Even in networks:
- **WIP is always additive**
- **cycle time must be computed using Little’s Law**
- **cycle times cannot be summed** if paths are parallel

---

## 5.3.3 Splitting Flows

### Definition
A splitting flow occurs when:
- one upstream workstation feeds
- multiple downstream branches

Each branch receives only a **fraction of the total flow**.

---

### Arrival Rate after Splitting
If a fraction p goes to branch i:
```
λ_i = p · λ
```

---

### Variability after Splitting
If splitting is random:
```
C_A,i² = C_D²
```

If splitting is deterministic:
```
C_A,i² < C_D²
```

Splitting **does not increase variability**, but redistributes it.

---

## 5.3.4 Merging Flows

### Definition
A merging flow occurs when:
- multiple upstream branches
- feed a single workstation

This is the **most critical configuration** for variability.

---

### Arrival Rate after Merging
For two merging flows:
```
λ = λ₁ + λ₂
```

---

### Variability after Merging
Arrival variability increases due to **superposition**.

Approximation:
```
C_A² ≈ (λ₁/λ)·C_A1² + (λ₂/λ)·C_A2²
```

If flows are independent:
- variability increases
- congestion is amplified

---

## 5.3.5 Example 1 – Network with Splitting and Merging

### System Description
- External arrivals: exponential
- One upstream workstation
- Two parallel branches
- One downstream merging workstation

Procedure:
1. analyze upstream workstation
2. compute departure variability
3. split flow into two branches
4. analyze each branch independently
5. merge flows
6. compute arrival variability at final workstation

---

## 5.3.6 Example 2 – Serial Line with G/G/1 Stations

This example revisits the **serial system** with general distributions,
highlighting how the same procedure applies to networks.

For each workstation:
1. compute utilization
2. compute CTQ using VUT
3. compute WIP
4. compute departure variability

Total system performance:
```
WIP_total = Σ WIP_i
CT_total = WIP_total / λ
```

---

## 5.3.7 Ordering Rule for Network Performance

When choosing the order of workstations:

> **Place workstations with lower variability upstream.**

Reason:
- variability propagates downstream,
- early variability affects all subsequent paths,
- merging amplifies variability.

This rule allows **qualitative decisions without calculations**.

---

## 5.3.8 Example 3 – Choosing the Order of Two Workstations

Two workstations:
- A: exponential processing time (C_S² = 1)
- B: general processing time with σ = 5 min, E[TS] = 20 min

Coefficient of variation:
```
C_B = 5 / 20 = 0.25  →  C_B² = 0.0625
```

Conclusion:
> **Process B should be placed before A.**

This minimizes variability propagation and improves system performance.

---

## 5.3.9 Key Takeaways
- Networks generalize serial systems.
- Splitting redistributes flow; merging amplifies variability.
- WIP is always additive; cycle times are not.
- Variability propagation dominates network performance.
- Ordering low-variability stations first is optimal.
